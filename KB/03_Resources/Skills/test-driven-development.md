# Test-Driven Development (TDD)

## 铁律

```
NO PRODUCTION CODE WITHOUT A FAILING TEST FIRST
```

## Red-Green-Refactor 循环

### RED - 写失败的测试
写一个最小的测试，展示应该发生什么。

### GREEN - 最少代码
写最简单的代码让测试通过。

### REFACTOR - 重构
清理代码，去除重复。

## 为什么顺序重要

**"我之后再写测试"** - 之后写的测试会立即通过。通过只能证明测试工作了，不能证明代码正确。

**"我已经手动测试了所有边界情况"** - 手动测试是随意的。你以为测试了全部，但可能没有记录，不能重跑。

## 常见借口

| 借口 | 现实 |
|------|------|
| "太简单不用测试" | 简单代码也会坏，测试只需30秒 |
| "之后测试" | 测试立即通过什么都证明不了 |
| "TDD 太死板" | TDD 是务实的，比调试快 |

## 验证清单

- [ ] 每个新函数/方法都有测试
- [ ] 每个测试在实现前都看到失败
- [ ] 写了最少量代码让每个测试通过
- [ ] 所有测试通过
- [ ] 输出干净（无错误、无警告）
